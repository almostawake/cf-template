# Architecture & Code Generation Guide

This document defines the technology stack and patterns to use when building features in this template. The tooling is installed and configured, but most patterns described here (API endpoints, database tables, tests, content collections) are not yet implemented — they are created as features require them. Do not assume files or directories exist unless you have checked. Always verify before importing.

## Platform

All apps target Cloudflare's edge platform. Every deployment uses Cloudflare Pages (not standalone Workers). Server-side API logic runs as Pages Functions, which are Workers under the hood but deployed and routed through the Pages pipeline.

### Services Available

| Service | Purpose |
|---|---|
| **Cloudflare Pages** | Static site hosting + Pages Functions for server-side logic |
| **D1** | Primary database (SQLite) |
| **R2** | File/blob storage (S3-compatible) |
| **Workers KV** | Session data, config, feature flags |
| **Durable Objects** | Reserved for real-time coordination only — do not use unless the feature specifically requires it |

Bindings for D1, KV, and R2 are declared in `wrangler.jsonc` and available locally via `wrangler dev`. Remote resources (databases, namespaces, buckets) are created on first deploy or manually via `wrangler` CLI as needed.

### Environments

Three environments managed via Wrangler:

- `dev` — local development via `npm run dev`
- `staging` — deployed as `[app-name]-staging` (via Git branch or manual deploy)
- `production` — deployed as `[app-name]` (via Git push to main or manual deploy)

Each environment has its own D1 database, KV namespace, and R2 bucket. Bindings are defined per-environment in `wrangler.jsonc`. Secrets go in `.dev.vars` locally, and are set via `wrangler secret put` for deployed environments.


## Frontend

### Framework: Astro

Astro is the site framework, using the `@astrojs/cloudflare` adapter for Cloudflare Pages deployment. Pages are static HTML by default. No JavaScript ships to the browser unless a component explicitly opts in via a `client:` directive.

**Target file structure:** Create directories and files as features require them. Not all of these exist yet — this is the intended layout.

```
src/
  pages/            # File-based routing — .astro pages and API endpoints
    index.astro
    api/            # REST endpoints — create when adding API features
      [resource].ts
  components/       # Reusable Astro and Svelte components
  layouts/          # Page layouts (base HTML shell, nav, footer)
  content/          # Markdown content collections (create when needed)
  lib/              # Shared utilities (see Organisation Rules below)
    db/
      schema.ts     # Drizzle schema — single source of truth
      index.ts      # Database helper (getDb function)
    auth.ts         # Auth helper (create only when auth is enabled)
    validation.ts   # Shared Zod schemas (create when adding API endpoints)
  styles/           # Global CSS (minimal — font imports and resets only)
public/             # Static assets (favicon, images not in content)
drizzle/
  migrations/       # Generated by drizzle-kit — do not hand-edit
tests/              # Vitest test files (create when adding testable code)
```

**Rules:**

- Pages are `.astro` files. They import layouts and components.
- Layouts define the HTML shell (`<html>`, `<head>`, `<body>`). Pages wrap content in a `<Layout>` component.
- When using static content, place it in `src/content/` as Markdown files using Astro content collections with a schema defined in `src/content.config.ts`.
- All pages render as static HTML at build time unless explicitly marked with `export const prerender = false` for server-side rendering.

### Interactive Components: Svelte

When a component needs client-side interactivity (user input, dynamic filtering, toggling state), use Svelte.

**Rules:**

- Svelte components live in `src/components/` with `.svelte` extension.
- Only use Svelte when the component genuinely needs client-side JavaScript. If it can be static HTML, write it as an `.astro` component instead.
- Hydrate Svelte islands with the minimum directive needed:
  - `client:load` — hydrate immediately (forms, above-the-fold interactive elements)
  - `client:visible` — hydrate when scrolled into view (below-fold widgets)
  - `client:idle` — hydrate when browser is idle (non-critical enhancements)
- Keep Svelte components small and focused. One component per interactive concern.
- Svelte components receive data via props. They do not fetch data themselves — pass data from the Astro page or a server endpoint.

### Styling: Tailwind CSS

All styling uses Tailwind utility classes applied directly in markup.

**Rules:**

- No separate CSS files per component. Styles live in the markup via Tailwind classes.
- Use Tailwind's design tokens (spacing scale, colour palette, typography scale) consistently. Do not use arbitrary values like `p-[13px]` unless there is no reasonable Tailwind token.
- Dark mode support is optional per-project. When used, apply `dark:` variants.
- Global base styles (if any) go in `src/styles/global.css` and are limited to resets or font imports.
- Astro components use `class="..."` for Tailwind. Svelte components also use `class="..."` on elements directly.


## Backend

### Language: TypeScript

All server-side code is TypeScript. Strict mode is enabled (see `tsconfig.json`).

**Cloudflare binding types:** The `@astrojs/cloudflare` adapter bridges `worker-configuration.d.ts` (generated by `npm run types`) into Astro's runtime — bindings are accessed via `locals.runtime.env` which is typed against the generated `Env` interface. Never hand-write the `Env` interface. Always regenerate with `npm run types` after changing bindings in `wrangler.jsonc`.

### API Design: REST

When adding server-side endpoints, place them in `src/pages/api/` following REST conventions. These are deployed as Pages Functions automatically. Create the `api/` directory when adding the first endpoint.

**File-to-route mapping:**

```
src/pages/api/users.ts       → GET/POST  /api/users
src/pages/api/users/[id].ts  → GET/PUT/DELETE /api/users/:id
```

**Endpoint structure:**

```typescript
import type { APIRoute } from "astro";
import { getDb } from "../../lib/db";
import { createUserSchema } from "../../lib/validation";

export const GET: APIRoute = async ({ params, locals }) => {
  const db = getDb(locals.runtime.env.DB);

  try {
    // ... query and return
    return new Response(JSON.stringify(data), {
      headers: { "Content-Type": "application/json" },
    });
  } catch (error) {
    return new Response(
      JSON.stringify({ message: "Internal server error" }),
      { status: 500 }
    );
  }
};

export const POST: APIRoute = async ({ request, locals }) => {
  try {
    const body = await request.json();
    const parsed = createUserSchema.safeParse(body);

    if (!parsed.success) {
      return new Response(
        JSON.stringify({ message: "Validation failed", errors: parsed.error.flatten() }),
        { status: 400 }
      );
    }

    const db = getDb(locals.runtime.env.DB);
    // ... insert and return
    return new Response(JSON.stringify(result), { status: 201 });
  } catch (error) {
    return new Response(
      JSON.stringify({ message: "Internal server error" }),
      { status: 500 }
    );
  }
};
```

**Rules:**

- One file per resource. HTTP methods are named exports (`GET`, `POST`, `PUT`, `DELETE`).
- Always validate request bodies with a Zod schema before passing to the database.
- Return appropriate HTTP status codes: 200 (ok), 201 (created), 400 (bad request), 401 (unauthorized), 404 (not found), 500 (server error).
- Return JSON for all API responses. Errors include a `message` field.
- Access Cloudflare bindings via `locals.runtime.env`, never via global variables.
- Every endpoint has a top-level try/catch. Never let an unhandled exception propagate.

### Validation: Zod

When API endpoints accept input, validate with Zod. Create `src/lib/validation.ts` when adding the first endpoint that needs validation.

**Rules:**

- Shared schemas live in `src/lib/validation.ts` or alongside the relevant feature in `src/lib/`.
- Each POST/PUT endpoint has a schema. Parse with `schema.safeParse(body)` and return 400 on failure with Zod's flattened error output.
- Zod schemas define API input shapes. Drizzle schemas define storage shapes. These are related but separate concerns.

### Database: D1 via Drizzle ORM

Use D1 as the primary database with Drizzle ORM for schema definition, query building, and migration management. The binding name is `DB`. The schema file and `getDb` helper are scaffolded but empty — add tables as features require them.

**Schema location:** `src/lib/db/schema.ts`

**Example schema:**

```typescript
import { sqliteTable, text, integer } from "drizzle-orm/sqlite-core";

export const users = sqliteTable("users", {
  id: text("id").primaryKey(),
  email: text("email").notNull().unique(),
  name: text("name").notNull(),
  plan: text("plan").notNull().default("free"),
  createdAt: integer("created_at", { mode: "timestamp" }).notNull(),
});
```

**Database helper:** `src/lib/db/index.ts` provides `getDb(d1)` — see the file for usage.

**Rules:**

- All tables are defined in `src/lib/db/schema.ts`. This is the single source of truth for the data model.
- Use Drizzle's query builder for all database operations. Do not write raw SQL strings.
- Generate migrations with `npm run db:generate`. Apply with `npm run db:migrate:local` (dev) or the appropriate environment script.
- When a document-style pattern fits better than strict relational columns, use a `text` column storing JSON and extract fields with Drizzle's SQL helpers or SQLite's `json_extract`.
- Every table has an `id` column as primary key. Use `crypto.randomUUID()` for ID generation.
- Timestamp columns use `integer` with `{ mode: "timestamp" }` storing Unix epochs.

### File Storage: R2

Use R2 for user-uploaded files, media assets, and any binary data when a feature requires file storage. The binding name is `STORAGE`.

**Rules:**

- Access R2 via `locals.runtime.env.STORAGE`.
- Store files with meaningful key paths: `uploads/{userId}/{filename}`.
- Store file metadata (original name, content type, size, upload date) as a row in D1, with the R2 key as a column. R2 is the blob store; D1 is the index.
- Generate presigned URLs for client-side downloads when the file shouldn't be publicly accessible.

### Session & Config: Workers KV

Use KV for data that is read frequently, written rarely, and can tolerate eventual consistency (up to 60 seconds). The binding name is `KV`.

**Rules:**

- Use KV for: session tokens, feature flags, cached config.
- Do not use KV as a primary data store. It is a cache/config layer.
- Keys follow the pattern `{namespace}:{identifier}` — e.g., `session:{token}`, `config:feature-flags`.
- Always set a TTL on session data: `await env.KV.put(key, value, { expirationTtl: 86400 })`.


## Authentication (Optional)

Not every project needs auth. When auth is required, use Firebase Auth with edge-compatible libraries. Do not include auth boilerplate in projects that don't need it.

### When auth is enabled

**Additional dependencies:** `firebase-auth-cloudflare-workers`, `firebase` (client SDK for browser-side login UI).

**Additional KV namespace:** Add a `PUBLIC_JWK_CACHE_KV` namespace to `wrangler.jsonc` for caching Firebase's public signing keys.

**Additional environment variables:** `FIREBASE_PROJECT_ID`, `PUBLIC_JWK_CACHE_KEY`.

**Auth flow:**

1. User signs in via Firebase client SDK in the browser (Svelte component).
2. Browser sends the Firebase ID token in the `Authorization: Bearer {token}` header with every API request.
3. Server-side API endpoint verifies the token using `firebase-auth-cloudflare-workers`.
4. If valid, the endpoint proceeds. If invalid, return 401.

**Auth helper location:** `src/lib/auth.ts`

```typescript
import { Auth, WorkersKVStoreSingle } from "firebase-auth-cloudflare-workers";

export async function verifyAuth(request: Request, env: Env) {
  const authorization = request.headers.get("Authorization");
  if (!authorization) return null;

  const token = authorization.replace(/Bearer\s+/i, "");
  const auth = Auth.getOrInitialize(
    env.FIREBASE_PROJECT_ID,
    WorkersKVStoreSingle.getOrInitialize(
      env.PUBLIC_JWK_CACHE_KEY,
      env.PUBLIC_JWK_CACHE_KV
    )
  );

  try {
    return await auth.verifyIdToken(token);
  } catch {
    return null;
  }
}
```

**Rules:**

- Never trust client-provided user data without verifying the ID token server-side first.
- The `verifyAuth` helper returns the decoded token (with `uid`, `email`, etc.) or `null`. Endpoints check for `null` and return 401.
- Firebase project config (project ID) is stored as an environment variable, not hardcoded.

### When auth is enabled, add to wrangler.jsonc:

```jsonc
// Add to top-level kv_namespaces array:
{ "binding": "PUBLIC_JWK_CACHE_KV", "id": "my-app-jwk-cache" }

// Add to staging kv_namespaces array:
{ "binding": "PUBLIC_JWK_CACHE_KV", "id": "my-app-jwk-cache-staging" }
```


## Testing

Use Vitest with `@cloudflare/vitest-pool-workers` for testing. This runs tests inside the Workers runtime with access to real local bindings (D1, KV, R2). The test tooling is installed and configured — create test files as you add testable code.

**Test location:** `tests/` directory at project root.

**Rules:**

- API endpoints get integration tests. Each endpoint file `src/pages/api/users.ts` should have a corresponding `tests/api/users.test.ts`.
- Tests run against local D1/KV/R2 bindings (not deployed resources).
- Test files follow the naming convention `*.test.ts`.
- Tests cover the happy path and at least one error case (bad input, missing auth, not found) per endpoint.
- Do not mock Cloudflare bindings — the vitest pool provides real local bindings. Mock only external services (Firebase Auth verification, third-party APIs).

**Example test:**

```typescript
import { describe, it, expect } from "vitest";

describe("GET /api/users", () => {
  it("returns a list of users", async () => {
    const response = await fetch("http://localhost/api/users");
    expect(response.status).toBe(200);
    const data = await response.json();
    expect(Array.isArray(data)).toBe(true);
  });
});
```

## Project Conventions

### File Naming

- Astro pages and layouts: `kebab-case.astro` (e.g., `about-us.astro`)
- Svelte components: `PascalCase.svelte` (e.g., `SearchWidget.svelte`)
- TypeScript modules: `kebab-case.ts` (e.g., `auth.ts`, `schema.ts`)
- API endpoints: `kebab-case.ts` matching the resource name
- Test files: `*.test.ts` matching the file they test

### Code Organisation

- `src/lib/` contains all shared logic: database helpers, auth, validation, utilities.
- `src/components/` contains UI components only — no business logic, no direct database access.
- `src/pages/api/` contains server endpoints — these orchestrate (call lib functions) but keep business logic in `src/lib/`.
- **Directory depth rule for `src/lib/`:** One level of subdirectory is allowed. `src/lib/db/schema.ts` is correct. `src/lib/db/queries/users.ts` is too deep — flatten to `src/lib/db/user-queries.ts` instead. The goal is fast navigation, not elaborate taxonomy.

### Error Handling

- API endpoints wrap logic in try/catch. Unexpected errors return 500 with a generic message — never expose internal details.
- Database errors and auth errors are caught and mapped to appropriate HTTP status codes.
- Validation errors return 400 with Zod's flattened error output.
- Never let an unhandled exception propagate. Every endpoint has a top-level try/catch.

### Dependencies

Do not add libraries for things Cloudflare or the platform provides natively (HTTP client, crypto, UUID generation, JSON handling). Check `package.json` for what is already installed.


## When to Escalate

These patterns are not covered by this template and require additional architectural decisions:

- **Real-time features** (chat, live updates, collaboration): requires Durable Objects with WebSocket support.
- **Background jobs** (email sending, image processing): requires Cloudflare Queues or scheduled Workers.
- **Search** (full-text search across content): requires either SQLite FTS5 in D1 or an external search service.
- **Multi-tenant data isolation**: requires per-tenant D1 databases or Durable Objects per tenant.
- **Scheduled tasks**: requires Cron Triggers on Workers.

Do not implement these patterns without extending this architecture document first.
